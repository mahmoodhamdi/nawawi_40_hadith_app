# Contributing | المساهمة

> صدقة جارية — every line you contribute is in your scale forever, إن شاء الله.

Thank you for considering a contribution. This is an open-source Islamic
app intended as ongoing charity. The bar for what we accept is **the
benefit of Muslim users** worldwide.

---

## Quick rules / القواعد السريعة

1. **No Arabic hadith text edits without scholarly review.** Even fixing
   what looks like a typo in `assets/json/40-hadith-nawawi.json` requires
   evidence from a canonical printed edition or sunnah.com.
2. **No new tracking, analytics, or telemetry.** Period. Not even
   "anonymous". The app's privacy promise is non-negotiable.
3. **No ads. No paywalls. No "premium" tiers.** The app must stay 100% free.
4. **Keep new permissions to a strict minimum.** If your PR adds an
   Android or iOS permission, explain *why* in the PR description.
5. **Speak the way an imam would speak to children:** be kind, be brief,
   be precise. No gamification language ("XP", "level up", "achievement
   unlocked"). Use "استمرارية", "مراجعة", "تذكير".

## Code style

- **Dart**: pass `flutter analyze --fatal-warnings --fatal-infos`.
- **Formatting**: `dart format lib/ test/ integration_test/`.
- **Tests required** for new cubits, services, and pure functions.
- **State**: BLoC/Cubit pattern only. Don't introduce Riverpod, Provider,
  GetX, etc. — the project already standardizes on `flutter_bloc`.
- **No new dependencies without justification.** Each package adds APK
  size, security surface, and license review. Common alternatives that
  *avoid* a new dep include:
  - URL launching → `share_plus` + clipboard fallback
  - File picking → text-paste dialog (for small payloads)
  - HTTP → the app is offline-first; you probably don't need network

## Content contributions

- **Translations of hadith text** must come from a named, scholarly
  source (e.g. published edition, sunnah.com curated translator).
  Auto-translation tools are not acceptable for hadith content.
- **Translations of UI strings** (in `assets/l10n/*.arb`) can come from
  any fluent Muslim contributor. Two-reviewer rule: a second native
  speaker must approve before merge.
- **New "topic" tags** in `assets/json/40-hadith-nawawi.json` follow the
  scheme in `.agent/build_topics.py`. Each new tag needs:
  - A stable English snake_case ID
  - Arabic + English display labels
  - Documentation of why it's a separate tag rather than a subset of an
    existing one.

## PR workflow

1. Fork → branch → PR against `main`.
2. **Branch name**: `feat/<thing>`, `fix/<thing>`, `docs/<thing>`,
   `content/<thing>`, `chore/<thing>`. No personal-name prefixes.
3. **Title**: [Conventional Commits](https://www.conventionalcommits.org/).
   Example: `feat(quiz): add timer-free MCQ session`.
4. **Body** of the PR must answer:
   - What does this change?
   - Why now? (link issue or describe motivation)
   - How was it tested? (which tests / smoke checks)
   - Does it change any permission, ad/tracking surface, or content?
5. **CI must be green** before review.
6. **At least one reviewer** for any code change.
7. **At least two reviewers** for any change touching the Arabic hadith
   text — one must be a native Arabic speaker.

## Commit hygiene

- One logical change per commit. If your PR has unrelated work, split it.
- Imperative mood, English: "add quiz mode", not "added" or "adds".
- Don't write commit messages that mention coding assistants or generic
  markers like "generated by" or "co-authored-by AI". The work is the
  work; we keep the history about the code, not the toolchain that
  helped write it.

## Local development

```bash
# Toolchain (one-time)
# Flutter 3.8+, Dart 3.8+, JDK 17 for Android builds

flutter pub get
flutter analyze --fatal-warnings --fatal-infos
flutter test
```

For visual changes, please run on a real device or emulator before
submitting — golden tests catch some regressions but not all.

## Reporting bugs

Open an issue at https://github.com/mahmoodhamdi/nawawi_40_hadith_app/issues

Include:
- App version (visible in Settings → About)
- Platform + OS version
- Steps to reproduce
- Expected vs. actual behavior

For **content issues** (mistakes in Arabic text, narrator name, citation),
please open the issue with the `content` label and a link to the canonical
source supporting your correction.

## Security

For security disclosures, open a private security advisory on GitHub
rather than a public issue. We will respond within 7 days.
